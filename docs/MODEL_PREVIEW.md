# Model Preview

Model Search includes a stable square preview surface and preview state reporting. Human, Demihuman, and Monster entries use the same managed software renderer. Native CharaView allocation remains disabled because the current APIs do not provide safe slot and texture ownership.

The UI and backend are separated through `IModelPreviewBackend`; `MainWindow` never accesses game memory. `ModelPreviewController` owns selection and lifetime management around `SoftwareModelPreviewBackend`. The backend consumes only bounded managed CPU meshes and exposes an immutable scene plus bounded camera values to the UI.

Selections are identified by ModelChara row, category, source, and source row rather than Model ID alone. A 200 ms debounce prevents rapidly scrolling the result list from creating every intermediate preview. Each selection has a generation, and superseded generations are released before the newest model is dispatched.

The controller releases its backend when Model Search is not visible, the window stops drawing, the client logs out, the territory changes, or the plugin is disposed. The latest selection is rescheduled after the UI and game session become active again. Backend selection, release, camera reset, and disposal exceptions are contained and written to diagnostics as `AM7102`/`AM7103` events.

Model Details reports preview preparation independently of the rendering backend. Human entries validate their in-memory appearance and resolve age-specific face and hair models plus race/gender equipment fallbacks. Monster and Demihuman entries list their resolved game asset paths and whether each file exists. This lets the backend consume validated inputs instead of constructing paths during rendering.

Human readiness is produced by `HumanPreviewDataBuilder`, not by null checks. It validates payload sizes, race/sex values, ModelChara and source identity, and exact agreement between the display payload and apply backing data before producing immutable preview input. Young and Old NPC faces and hair use their body-type model code. NPC-only Face values retain their exact ID first and then try the normalized lower two-digit model ID, which resolves cases such as Au Ra Young Face `102` to `f0002`. Young body and equipment resolution follows the PBD ancestry used by the game: the race-specific Young code, shared Young `c0104`, then shared Human `c0101`. Shared ancestor parts are deformed to the selected Young skeleton before scene assembly.

Hair materials use the character mask sampler and accept `_s.tex` in addition to the newer mask suffixes. Human previews use `HumanAppearance.Customize` and the `human.cmp` hair palette. Demihuman previews instead read the MTRL `g_DiffuseColor`, because special models can store their fixed hair color in the material itself and can carry hair geometry in non-Head parts such as Hands. Human `skin.shpk` materials combine their diffuse texture, normal-map skin influence, and the selected race/clan/gender SkinColor from `human.cmp`.

For all three categories, the settled selection is inspected through a bounded MDL V5/V6 preview parser. It reads only the file header, vertex declarations, High LOD mesh records, position streams, and 16-bit indices, avoiding unrelated bone-table data. Model Details displays aggregate Mesh, vertex, index, triangle, LOD, combined bounds, and computed Auto Frame distance. Missing or invalid parts produce a partial/failed geometry result without discarding valid parts. The software scene samples at most 6,000 non-degenerate triangles, uses deterministic colors and flat lighting, and performs depth-sorted orthographic projection into the ImGui canvas. It allocates no native game object, CharaView slot, render target, shader, texture, or unmanaged GPU resource.
